# Extraction Verification

This step verifies that an input document can be safely round-tripped through the extraction and merging process. It extracts the raw document, merges it, then extracts again — comparing the events from both extractions to detect discrepancies. Any differences indicate potential problems with merging files back after translation. This is especially useful as a first pipeline step when working with newly developed or less-tested filters.

**Takes:** Raw document. **Sends:** Raw document (unmodified).

## Parameters

#### Perform Extraction Verification (`stepEnabled`)

Master switch for this step. When disabled, the input document passes through to the next step without any verification processing. Useful for temporarily bypassing verification in a pipeline without removing the step.

#### Verify All Events (`allEvents`)

Controls the scope of event comparison between the two extractions.

- **Enabled**: Every event generated by the filter is compared (text units, document parts, skeleton, etc.)
- **Disabled**: Only text unit events are compared

Verifying only text units is often sufficient — if the merge introduced errors, text units will typically differ between the two extractions. Disable this to reduce processing time on large documents.

> **Note:** Verifying only text units is usually a reliable indicator of merge correctness, since merge errors typically manifest as text unit differences.

#### Compare Skeleton (`compareSkeleton`)

Controls whether skeleton parts of verified events are included in the comparison. Skeletons represent the non-extractable portions of the input document (markup, formatting, structural elements surrounding translatable content).

Comparing skeletons provides a more thorough verification but may flag differences that don't affect translatable content.

#### Maximum Warnings Per Document (`limit`)

Caps the number of warning messages generated per document. Once this limit is reached, the step either moves on to the next document or interrupts the pipeline (depending on the **interrupt** setting).

#### Interrupt on Warning Limit (`interrupt`)

Controls whether the entire pipeline stops when the warning limit is reached for a document.

- **Enabled**: The pipeline is interrupted (halted) when the maximum number of warnings is reached
- **Disabled**: The step moves on to the next document after reaching the limit, allowing the pipeline to continue

Only meaningful when the warning limit (set by `limit`) is actually reached.

## Limitations

- Support for multilingual documents (XLIFF, TMX, PO, TTX, etc.) is **BETA**.

## Notes

- The step performs a full extract → merge → extract cycle internally, so processing time is roughly 3x a single extraction pass.
- The raw document is sent to the next step **unmodified** — the verification is purely a side-effect check.
- Requires both source and target language to be set in the pipeline for the merge step.

## Examples

### Pipeline guard for untested filters

Place the Extraction Verification step as the **first step** in a pipeline that uses a newly developed or less-tested filter. Any warnings indicate that the filter's merge implementation may produce incorrect output after translation.

```yaml
steps:
  - id: extraction-verification
    params:
      stepEnabled: true
      allEvents: true
      compareSkeleton: true
      limit: 10
      interrupt: true
```

### Lightweight text-unit-only check

For faster processing on large document sets, verify only text units and allow the pipeline to continue past warnings.

```yaml
steps:
  - id: extraction-verification
    params:
      stepEnabled: true
      allEvents: false
      compareSkeleton: false
      limit: 20
      interrupt: false
```
